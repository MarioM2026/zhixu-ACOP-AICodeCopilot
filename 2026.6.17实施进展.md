## ⭐ 核心价值主张

> **知墟的终极目标不是"监控 AI"，而是"让 AI 自我进化"**

知墟不仅是一个监控工具，更是一个 **AI 进化引擎**。我们的核心价值：

| 传统监控工具 | 知墟的差异化 |
|-------------|-------------|
| 被动展示数据 | **主动执行规则**：Token 超限自动告警、上下文溢出自动清理 |
| 人工分析问题 | **自动注入提示**：高频错误模式自动识别并注入优化 Prompt |
| 单次告警 | **持续闭环**：告警 → 执行 → 验证效果 → 调整规则 |
| 监控工具 | **AI 进化引擎**：让 AI 从错误中学习，越来越聪明 |

**反馈闭环的价值链：**
```
采集数据 → 分析模式 → 执行规则 → 验证效果 → 优化规则 → AI 进化
```

## 一、项目概述

知墟（ZhiXu - AI Code Copilot Observability Platform）是一个统一的 AI 编程助手观测与优化平台，支持 Trae、Claude Code、Cursor 等工具的数据采集、监控和自动优化。

**核心目标（按重要性排序）：**

| 优先级 | 目标 | 说明 |
|--------|------|------|
| **P0 最核心** | ⚙️ 规则引擎自动执行 | 定时扫描事件数据，自动触发规则，执行上下文清理/告警发送/Prompt 注入 |
| **P0 核心差异化** | 🔄 错误反馈闭环 | 采集错误模式 → 分析高频错误 → 注入优化提示，让 AI 越来越聪明 |
| P1 | 📊 统一数据采集 | 实时采集所有接入的 AI 编程工具数据，遵循 OTel GenAI 语义规范 |
| P2 | 📈 实时可视化监控 | 提供直观的运营指标看板 |
| P3 | 🔌 异构工具兼容 | 支持多种主流 AI 开发工具无缝接入 |

## 六、技术债务与风险

### 6.1 技术债务

1. **内存存储**：所有数据存储在内存中，进程重启后数据丢失
   - 影响：看板数据每次都是新生成的模拟数据
   - 风险：中 — 不影响演示，但不能用于生产
   - 修复：接入 SQLite → PostgreSQL

2. **模拟数据**：`dashboardService.ts` 中使用伪随机生成
   - 影响：展示的数据不反映真实使用情况
   - 风险：中 — 开发期可以接受，但需尽早接入真实数据源
   - 修复：开发 Trae/Claude Code 适配器的真实采集能力

3. **前端请求结构不一致**：前端对部分 API 返回用 `response.data?.success`，实际后端返回为 `{ success, message }`
   - 影响：已修复，但需注意新增接口时保持一致
   - 风险：低

### 6.2 风险点

| 风险 | 级别 | 说明 |
|------|------|------|
| Trae API 变更 | 中 | Trae 不保证 API 稳定，可能影响适配器 |
| 缺少生产数据库 | 高 | 当前无持久化存储，生产部署前必须解决 |
| 无用户认证 | 高 | 所有数据无隔离，不适合多人使用 |
| 无错误恢复机制 | 中 | 服务崩溃后无自动恢复和重试机制 |
