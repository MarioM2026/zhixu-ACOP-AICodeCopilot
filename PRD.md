## 产品概述

### 项目名称

知墟 - AI 编程助手观测与优化平台 (ZhiXu - AI Code Copilot Observability Platform)

### 核心理念

通过"数据驱动"的**可观测性**与**反馈闭环**，将 AI 编程工具从"盲盒"转化为可量化、可优化、可追溯的透明系统，使 Trae、Cursor、Claude Code 等工具在统一平台上持续进化。

### 产品定位

打造统一的 AI 编程助手观测与优化平台，为技术团队提供：
- 多工具并行使用时的统一数据视图
- 精细化的成本归因与控制能力
- 智能化的性能优化与反馈闭环
- 标准化的跨工具扩展能力

### 为什么规则执行和反馈闭环是核心？

| 传统监控 | 知墟的差异化 |
|---------|-------------|
| 被动展示数据 | **主动执行规则**：Token 超限自动告警、上下文溢出自动清理 |
| 人工分析问题 | **自动注入提示**：高频错误模式自动识别并注入优化 Prompt |
| 单次告警 | **持续闭环**：告警 → 执行 → 验证效果 → 调整规则 |
| 监控工具 | **AI 进化引擎**：让 AI 从错误中学习，越来越聪明 |

### 反馈闭环的价值链

```
1️⃣ 采集数据
   └── Trae/Claude Code 会话数据（Token、错误、延迟）

2️⃣ 分析模式
   └── 识别高频错误类型、上下文溢出趋势、性能瓶颈

3️⃣ 执行规则
   └── 自动清理上下文 / 发送告警 / 注入优化提示

4️⃣ 验证效果
   └── 记录执行结果，评估错误率是否下降

5️⃣ 优化规则
   └── 根据效果数据调整规则参数，形成数据飞轮

6️⃣ AI 进化
   └── 错误模式减少 → AI 表现越来越好 → 开发效率提升
```

### 量化价值

| 指标 | 价值 |
|------|------|
| 错误模式减少 | 高频语法错误减少 30%+ |
| 上下文利用率 | Token 利用率提升 20%+ |
| 开发效率 | 每日节省 30 分钟重复修正时间 |
| AI 进化 | 随着使用，AI 越来越懂你的代码风格 |

## 产品目标与范围

### 核心目标（按重要性排序）

1. **⚙️ 规则引擎自动执行（最核心）**
   - 定时扫描事件数据，自动触发规则
   - 执行动作：上下文清理、告警发送、Prompt 注入
   - 这是知墟区别于普通监控工具的关键能力

2. **🔄 错误反馈闭环（核心差异化）**
   - 采集错误模式 → 分析高频错误 → 注入优化提示
   - 让 AI 从错误中学习，越来越聪明
   - 形成数据飞轮，持续优化

3. **📊 统一数据采集**
   - 实时采集所有接入的 AI 编程工具数据
   - 遵循 OTel GenAI 语义规范

4. **📈 实时可视化监控**
   - 提供直观的运营指标看板
   - 支持多维度成本/性能评估

5. **🔌 异构工具兼容**
   - 支持多种主流 AI 开发工具无缝接入
   - 统一数据模型，跨工具分析

## 业务价值与量化指标

### 痛点与优化

| 维度 | 优化前痛点 | 优化后价值 | 核心 KPI |
|------|-----------|-----------|---------|
| **可观测性** | 多工具并行，状态跟踪困难 | 统一视图，全链路追踪 | 问题定位时间缩短率 > 50% |
| **成本控制** | Token 消耗缺乏精细化拆分 | 按模型/工具的成本归属分析 | 单位任务 Token 成本降低 > 20% |
| **开发效率** | 上下文传递不完整导致协作错误 | 上下文健康度监控与预警 | AI 代码接受率提升、手工修正次数下降 |
| **自优化能力** | 缺乏反馈闭环，模型迭代不稳定 | 基于数据驱动的自动优化 | 自动化优化任务执行成功率 > 90% |

- **市场机会**: 企业级可观测性平台处于空白期
- **竞争优势**: 首个开源统一的跨工具观测平台

### 核心组件说明

#### L1 统一数据总线层

| 组件 | 技术选型 | 职责 |
|------|---------|------|
| OpenTelemetry Collector | OTel Collector | 统一接收、处理、导出遥测数据 |
| Agent 适配器 | 自研 | 连接异构 AI 编程工具，标准化数据格式 |

#### L2 数据处理层

| 组件 | 技术选型 | 职责 |
|------|---------|------|
| 规则引擎 | Python/Drools | 定义和执行数据处理规则 |
| 流处理 | Apache Flink / 本地脚本 | 实时数据清洗与聚合 |
| 双层 Loop | 自研 | 任务级优化 + 模型级优化 |

#### L3 应用与交付层

| 组件 | 技术选型 | 职责 |
|------|---------|------|
| 可视化看板 | Grafana + React | 运营指标展示、实时监控 |
| REST API | FastAPI / Express | 封装核心指标查询 |
| 告警通知 | 钉钉/飞书/邮件 SDK | 多通道告警推送 |

#### L4 反馈与执行层

| 组件 | 技术选型 | 职责 |
|------|---------|------|
| 自动化优化 | 自研引擎 | 基于规则的自动优化执行 |
| 全链路闭环 | TaskStateMachine | 追踪优化任务执行状态 |

### 数据模型 (OTel GenAI 语义规范)

```typescript
interface AICodeEvent {
  // 基础信息
  sessionId: string;           // 会话 ID
  traceId: string;              // 链路 ID
  timestamp: number;            // 时间戳

  // 工具信息
  tool: 'trae' | 'claude_code' | 'cursor';  // 工具类型
  modelId: string;              // 模型标识

  // Token 消耗
  inputTokens: number;         // 输入 Token 数
  outputTokens: number;        // 输出 Token 数
  totalTokens: number;         // 总 Token 数

  // 性能指标
  latency: number;             // 响应延迟 (ms)
  ttft?: number;               // 首 Token 时间 (ms)

  // 质量指标
  errorType?: string;          // 错误类型
  codeAcceptance?: boolean;    // 代码接受状态
  contextOverflow?: boolean;   // 上下文溢出标记

  // 成本归因
  cost: number;                // 本次调用成本
  cost_attribution: string;   // 成本归属 (项目/用户/部门)
}
```

## 核心功能模块

### A. 统一数据采集与标准化

#### 功能描述

实时监控所有接入的 AI 编程工具，通过 OpenTelemetry 标准协议输出统一格式数据。

#### 功能列表

| 功能点 | 描述 | 优先级 |
|--------|------|--------|
| OTEL 数据接收 | 接收来自各工具的 Trace/Log/Metric 数据 | P0 |
| 工具协议适配 | 不同工具的数据格式统一转换 | P0 |
| 数据脱敏 | API Key、敏感信息自动过滤 | P1 |
| 断点续传 | 网络中断时的数据缓存与重传 | P1 |

#### 用户故事

> 作为开发者，我希望启动知墟后能自动采集 Trae 的会话数据，无需手动配置，即可看到完整的开发活动记录。

### B. 可视化监控看板

#### 功能描述

提供直观的运营指标总览，支持实时运行状态追踪、多维度成本/性能评估、错误拓扑定位。

#### 功能列表

| 功能点 | 描述 | 优先级 |
|--------|------|--------|
| 运营总览 | 展示核心 KPI 指标卡片 | P0 |
| Token 消耗趋势 | 按日/周/月展示 Token 消耗曲线 | P0 |
| 错误类型分布图 | 环形图，按错误类型分组 | P0 |
| 工具调用排行 | 表格展示各工具使用频率 | P1 |
| 会话详情追踪 | 下钻查看单会话完整链路 | P1 |
| 实时监控大屏 | 全屏实时监控模式 | P2 |

#### 核心图表设计

1. **Token 消耗趋势图**: 折线图 + 面积图，展示日 Token 消耗趋势
2. **错误类型分布图**: 环形图，按错误类型分组
3. **工具使用热力图**: 日历热力图，展示每日各工具使用情况
4. **模型调用占比**: 堆叠柱状图，展示不同模型调用占比

### C. 智能自动优化引擎

#### 功能描述

| 功能点 | 描述 | 优先级 |

| 上下文清理 | Token 占用超阈值时自动清理 | P0 |
| Token 限流告警 | 单日/单次 Token 超预算时通知 | P0 |

| 错误模式注入 | 检测高频错误后注入优化提示 | P1 |
| 效果追踪 | 记录优化动作并评估效果 | P1 |

## 分阶段实施计划

### MVP 定位

2~4 周内快速打通 Trae IDE 的数据采集 → 基础看板展示 → 初阶反馈闭环，并基于 OpenTelemetry 标准架构，为 Cursor/Claude Code 等工具平滑扩展打好基础。

### 技术选型 (MVP)

- **后端**: Node.js 18+ / Python
- **数据采集**: OpenTelemetry Collector
- **可视化**: Grafana + Prometheus
- **前端**: React 轻量看板
- **数据库**: SQLite (前期) → PostgreSQL

**基础规则配置**:

  - name: context_cleanup
    condition: "token_usage > 0.8"
    action: "clear_context"
    priority: high

  - name: token_budget_alert
    condition: "daily_token > budget_threshold"
    action: "send_notification"
    channels: ["dingtalk", "email"]

  - name: error_pattern_injection

## 实施建议

### 关键成功要素

1. **先跑通，再优化**
   - 优先完成阶段二 + 三，快速获得可见性
   - 再逐步增加自动化闭环

2. **数据隐私优先**
   - 所有采集默认本地存储
   - 敏感信息脱敏（API Key、个人身份信息）

3. **低侵入维护**
   - 全程采用透明代理 / OTel 方案
   - 不修改 Trae 内核，避免版本升级冲突

4. **标准化先行**
   - 早期就采用 OTel 数据结构
   - 后续接入 Cursor、CodeGeeX 时无需重构

### 风险与应对

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|---------|
| Trae API 变更 | 中 | 高 | 预留适配器抽象层，监控官方动态 |
| OTel 标准演进 | 低 | 中 | 保持组件松耦合，易于升级 |
| 数据隐私合规 | 低 | 高 | 本地优先，遵循 GDPR 等法规 |
